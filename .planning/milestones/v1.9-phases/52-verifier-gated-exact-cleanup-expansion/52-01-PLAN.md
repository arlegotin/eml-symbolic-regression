<objective>
Expand the target-free repair core from selected-only cleanup to bounded candidate-pool cleanup.

Purpose: REP-01 through REP-03 require cleanup to inspect more exact candidates and subtree-shaped alternatives without changing the verifier-owned recovery contract or mutating optimizer-selected/fallback manifests.

Output: `RepairConfig.expanded_candidate_pool()`, repair report provenance for candidate roots, global exact-AST dedup across roots, and focused unit tests in `tests/test_repair.py`.
</objective>

<scope>
In scope:
- `src/eml_symbolic_regression/repair.py` cleanup logic and repair metadata only.
- `tests/test_repair.py` regression coverage for the new cleanup behavior.
- Backward-compatible defaults for existing direct callers of `RepairConfig()`.

Out of scope:
- Benchmark runner wiring, built-in suites, campaign/report columns, generated artifacts, and docs. Those are handled by 52-02 and 52-03.
- Changes to `FitResult`, `ExactCandidate`, `SoftEMLTree`, `expand_snap_neighborhood()`, optimizer selection, verifier semantics, or target-aware perturbed repair.
- Target-aware cleanup for ordinary target-free repair.
</scope>

<risks>
- Verifier cost can grow if duplicate roots or variants are not deduplicated globally; mitigate by exact-AST root and variant keys before verifier calls.
- Adding report fields can break direct test construction if defaults are missing; mitigate by defaulting every new field.
- Ranking could weaken verifier ownership if source-order tie-breakers come before verification errors; keep `_cleanup_ranking_key()` first.
- Cross-root subtree grafting would exceed current provenance. Use only serialized alternatives attached to each candidate root.
</risks>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Optimizer candidate pool -> repair cleanup | Serialized exact candidates and slot alternatives enter cleanup ranking. |
| Repair cleanup -> verifier | Deduplicated exact AST variants are evaluated for recovered/not-recovered status. |
| Repair report -> benchmark artifacts | Repair provenance is serialized for later aggregation and docs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-01-01 | Tampering | candidate-root selection | mitigate | Deduplicate roots by exact AST and never mutate selected/fallback candidate objects or manifests. |
| T-52-01-02 | Repudiation | repair provenance | mitigate | Serialize candidate root source/id, root counts, variant counts, and accepted root metadata. |
| T-52-01-03 | Denial of Service | expanded cleanup | mitigate | Keep larger settings bounded through `RepairConfig.expanded_candidate_pool()` and globally deduplicate variants before verification. |
| T-52-01-04 | Information Integrity | recovered status | mitigate | Accept only verifier status `recovered`; failed variants never serialize `repaired_ast`. |
</threat_model>
