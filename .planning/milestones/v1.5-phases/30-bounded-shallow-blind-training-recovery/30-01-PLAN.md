<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| suite contract -> benchmark expansion | Built-in or JSON suite metadata becomes executable proof runs. |
| proof target inventory -> claim matrix | Formula/case IDs can inflate paper claim coverage if not cross-validated. |
| exact shape evidence -> suite bound | Compiler helper evidence can be mistaken for blind training proof. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-01 | Tampering | `BenchmarkCase.validate` / `BenchmarkSuite.validate` | mitigate | Keep fail-closed claim, case ID, start mode, training mode, threshold, seed, and budget validation; tests reject non-blind shallow proof cases. |
| T-30-02 | Repudiation | `tests/test_shallow_scaled_exponential_contract.py` | mitigate | Record depth, node count, constants, and verifier status for every scaled target so the bound change is auditable. |
| T-30-03 | Information Disclosure | `proof_dataset_manifest` for new targets | accept | Manifests continue to omit raw arrays and expose only formulas, domains, hashes, and provenance; no secrets or private data are introduced. |
| T-30-04 | Tampering | `scaled_exponential_expr` | mitigate | Tests explicitly assert this helper is shape evidence only; later proof runs must still be blind `fit_eml_tree` executions with derived `blind_training_recovered` evidence. |
</threat_model>
