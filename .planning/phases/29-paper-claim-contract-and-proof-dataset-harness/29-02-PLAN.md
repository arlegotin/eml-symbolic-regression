<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| suite JSON -> benchmark validation | User-supplied suite files can try to attach unknown claims, weaker thresholds, or spoofed evidence classes. |
| benchmark execution -> run artifact | Run artifacts become proof evidence consumed by campaign/report code. |
| aggregate evidence -> claim status | Aggregates can accidentally promote compile/catalog success into training proof. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-02-01 | Tampering | `BenchmarkCase.from_mapping` / `validate` | mitigate | Validate known claim IDs, required threshold IDs, valid training modes, positive budgets, and reject caller-supplied evidence class. |
| T-29-02-02 | Repudiation | `_base_run_payload` | mitigate | Write code version, claim ID, training mode, threshold, budget, dataset manifest, and formula provenance into every artifact. |
| T-29-02-03 | Elevation of Privilege | `aggregate_evidence` | mitigate | Evaluate bounded 100% thresholds only from allowed verifier-owned training evidence classes, never from raw `claim_status` alone. |
| T-29-02-04 | Denial of Service | proof suites | accept | Phase 29 adds metadata and small tests only; expensive training recovery remains later phases and existing filters can run subsets. |
</threat_model>
